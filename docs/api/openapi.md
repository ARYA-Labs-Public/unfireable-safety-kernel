# OpenAPI contract

The Safety Kernel HTTP surface is defined by a single OpenAPI 3.1
document. The contract — not the source code — is the source of truth.
Every Rust handler, every client SDK, and every middleware ships against
it. If the YAML and the binary disagree, the YAML wins and the binary
is wrong.

## Where it lives

```
contracts/openapi/safety_kernel.yaml
```

Pin to this path in your tooling. The file is hand-authored, reviewed
on every change, and tagged with the kernel release version.

## Viewing it locally

Render the spec in a browser without committing to any particular doc
stack.

**Redocly (one-shot preview):**

```bash
npx @redocly/cli preview-docs contracts/openapi/safety_kernel.yaml
# Opens http://127.0.0.1:8080
```

**Swagger UI via docker:**

```bash
docker run --rm -p 8088:8080 \
  -e SWAGGER_JSON=/spec/safety_kernel.yaml \
  -v "$PWD/contracts/openapi:/spec" \
  swaggerapi/swagger-ui
# Opens http://127.0.0.1:8088
```

Either renderer is fine. Neither is endorsed; pick whichever fits your
existing docs pipeline.

## Endpoint summary

The table below is generated by reading the spec, not by hand-writing.
Re-derive it whenever the YAML changes.

| Method | Path | Operation ID | Summary |
|---|---|---|---|
| GET  | `/health` | `getLegacyHealth` | Health check (legacy path) |
| GET  | `/kernel/v1/health` | `getHealth` | Health check |
| GET  | `/kernel/v1/public_key` | `getPublicKey` | Get kernel public key for token verification |
| POST | `/kernel/v1/authorize` | `postAuthorize` | Authorize a sensitive action (signed token) |
| POST | `/kernel/v1/approvals/{item_id}/approve` | `postApprove` | Sign an approval decision |
| POST | `/kernel/v1/approvals/{item_id}/reject` | `postReject` | Sign a rejection decision |
| POST | `/policy/module/authorize` | `policyModuleAuthorize` | Authorize a module-load or code-execution event |
| POST | `/policy/module/register` | `policyModuleRegister` | *(draft)* Register a module path with required-pattern set |
| POST | `/policy/audit-event` | `policyAuditEvent` | *(draft)* Surface a non-decision policy audit event |
| GET  | `/policy/module/{module_path}/status` | `policyModuleStatus` | *(draft)* Query module registration and recent decisions |

The four endpoints marked *(draft)* are listed in the contract so their
URLs are reserved, but they respond `501 Not Implemented` in the
reference binary. Do not implement clients against draft shapes — the
request/response schemas are subject to change.

## Versioning policy

- The OpenAPI document carries an `info.version` field. It follows
  semantic versioning.
- **Patch** bumps: documentation-only edits, added optional fields,
  added endpoints, added enum values that don't change existing
  responses.
- **Minor** bumps: backward-compatible additions to the surface
  (new optional request fields, new optional response fields).
- **Major** bumps: any breaking change — removed endpoint, removed
  field, changed type, narrowed enum, changed semantics, changed
  status-code class for the same scenario.
- The path-prefix `/kernel/v1/...` is the major-version boundary for
  the kernel endpoints. A v2 surface would live under `/kernel/v2/...`
  and run side-by-side until v1 is retired.

The contract changes through pull requests. Schema diffs against the
last released version are part of the review.

## Generating clients

The contract is plain OpenAPI 3.1 — any conforming generator works. A
common starting point:

```bash
npx @openapitools/openapi-generator-cli generate \
  -i contracts/openapi/safety_kernel.yaml \
  -g <your-target-language> \
  -o ./generated-client
```

Supported targets include Go, Java, Kotlin, Python, TypeScript, and
many others. No particular generator is endorsed — pick whichever
matches your stack's existing conventions. The reference Rust client
in this repo is hand-written rather than generated, because the
circuit-breaker semantics matter more than transport boilerplate.

## Response signatures

The kernel signs responses that grant authority. Plain JSON is enough
for everything else.

- **ALLOW responses** (`/kernel/v1/authorize`, `/kernel/v1/approvals/*`,
  and ALLOW outcomes from `/policy/module/authorize`) carry an
  Ed25519-signed compact JWT in the `token` field plus its
  `token_sha256` fingerprint. Clients MUST verify the signature against
  the public key from `/kernel/v1/public_key` (or a pinned copy of it)
  before trusting the decision.
- **DENY responses** from `/policy/module/authorize` are *also* signed
  — DENY is itself an authoritative decision and is logged. Clients
  that care about audit can persist the signed DENY for later review.
- **Error responses** (`401`, `403`, `422`, `5xx`) are plain JSON with
  no signature. They are not authoritative decisions. A missing or
  unparseable response on a gated call MUST be treated as DENY.
