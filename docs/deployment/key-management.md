# Safety Kernel — signing-key management (`KERNEL_KEY_BACKEND`)

The Safety Kernel signs every authorization decision with an Ed25519
key. **Where that key comes from** is selected by the
`KERNEL_KEY_BACKEND` environment variable. This is the Step-14R key
management surface of the Phase-4 self-hosted deployment (ARY-1886).

> **One sentence:** in dev/staging the seed may come from an env var; in
> **production the env var is forbidden** and the seed must come from a
> managed secret store (GCP today; AWS/Azure/HSM/TPM tracked below).

## Backends

| `KERNEL_KEY_BACKEND` | Status | Where the seed lives | Tracking |
|----------------------|--------|----------------------|----------|
| `env` (default)      | ✅ implemented | `QORCH_KERNEL_SIGNING_KEY_B64` env var. **Blocked when `QORCH_ENV=prod`.** | — |
| `gcp`                | ✅ implemented | GCP Secret Manager, fetched at boot via metadata-server ADC | — |
| `aws`                | ⏳ not yet — fails closed | AWS Secrets Manager | [ARY-2518](https://linear.app/aryalabs/issue/ARY-2518) |
| `azure`              | ⏳ not yet — fails closed | Azure Key Vault | [ARY-2519](https://linear.app/aryalabs/issue/ARY-2519) |
| `pkcs11`             | ⏳ not yet — fails closed | PKCS#11 HSM (signs in-token) | [ARY-2520](https://linear.app/aryalabs/issue/ARY-2520) |
| `tpm`                | ⏳ not yet — fails closed | TPM 2.0 (sealed seed) | [ARY-2521](https://linear.app/aryalabs/issue/ARY-2521) |

Selecting an unimplemented backend **fails closed at startup** with an
error naming this document — it never silently falls back to the env
var. An unrecognized backend name is also a hard startup error.

## Generating a seed — `safety-kernel-keygen`

A fresh Ed25519 signing seed is generated by the `safety-kernel-keygen`
binary (built alongside the kernel). It prints the 32-byte seed as a
base64url (no-pad) string on **stdout**; the matching public key and its
SHA-256 fingerprint go to **stderr** for verification:

```console
$ safety-kernel-keygen
public_key_b64url:   GR_np-WYATEMD_vEQd1VvTUG4qbkzkfPEpOYUwNoMx4
public_key_fp_sha256: 3408ba12712d4a59d9e16c65ef53074ce496ac111d62c47741570cc87c84f2ed
BUqB3wFQAadM16O_Mxj_zlVliSCEwyOpVKZEPBtdkX8     # <- the seed (stdout)
```

The kernel's runtime identity only ever **reads** the seed. Generating
and **storing** it is an operator / Terraform step — this keeps the
kernel's service account read-only (least privilege). The published
public-key fingerprint lets you confirm later that the kernel loaded the
seed you intended (it matches `GET /kernel/v1/public_key`).

## `env` backend (dev / staging only)

```bash
export KERNEL_KEY_BACKEND=env          # or leave unset; env is the default
export QORCH_KERNEL_SIGNING_KEY_B64="$(safety-kernel-keygen 2>/dev/null)"
```

**Production guard.** When `QORCH_ENV=prod` (or `production`),
`KERNEL_KEY_BACKEND=env` is refused at startup:

```
fail-closed: KERNEL_KEY_BACKEND=env is forbidden when QORCH_ENV=prod.
Set KERNEL_KEY_BACKEND to a managed backend (gcp|aws|azure|pkcs11|tpm)
so the Ed25519 seed is not held in the process environment.
```

This stops the raw seed from sitting in `/proc/<pid>/environ` or a core
dump on a production host.

> **Roadmap note:** ARY-1886 AC also calls for a *compile-time* guard
> (a build feature that strips the `env` arm entirely from prod images),
> in addition to this runtime check. The runtime guard is the
> load-bearing control and ships here; the compile-time feature-gate is a
> follow-up hardening tracked on ARY-1886.

## `gcp` backend (production-ready)

Fetches the seed from **GCP Secret Manager** at boot using an OAuth2
access token from the GCE/GKE **metadata server** (attached service
account / Workload Identity). This is the canonical ADC path for
GCP-hosted workloads (GCE, GKE, Cloud Run). Non-GCE ADC (service-account
JSON key files) is **not** supported in this slice.

### 1. Provision the secret (operator, once)

```bash
PROJECT=my-project
SECRET=safety-kernel-signing-key

# Create the (empty) secret shell — Terraform usually owns this.
gcloud secrets create "$SECRET" --project="$PROJECT" --replication-policy=automatic

# Generate a seed and store it as the first version. The seed never
# touches disk.
safety-kernel-keygen 2>/dev/null | \
  gcloud secrets versions add "$SECRET" --project="$PROJECT" --data-file=-
```

### 2. Grant the kernel's service account read-only access

```bash
SA=safety-kernel@my-project.iam.gserviceaccount.com   # the kernel's runtime SA / Workload Identity

gcloud secrets add-iam-policy-binding "$SECRET" --project="$PROJECT" \
  --member="serviceAccount:$SA" \
  --role="roles/secretmanager.secretAccessor"
```

Bind `secretAccessor` **on the single secret**, not project-wide. The
kernel needs no write/list/admin permission.

### 3. Configure the kernel

```bash
export KERNEL_KEY_BACKEND=gcp
export KERNEL_KEY_GCP_PROJECT=my-project
export KERNEL_KEY_GCP_SECRET=safety-kernel-signing-key
export KERNEL_KEY_GCP_SECRET_VERSION=latest   # optional; default "latest"
# QORCH_KERNEL_SIGNING_KEY_B64 is NOT set under the gcp backend.
```

On boot the kernel logs `signing-key backend resolved backend=gcp` and
serves the matching public key at `GET /kernel/v1/public_key`. If the
fetch fails (missing IAM, missing secret, no metadata server) the kernel
**refuses to start** — a kernel without a verified signing identity is
not a kernel.

### Verifying end-to-end

`crates/services/safety-kernel/tests/gcp_key_backend_live.rs` is an
`#[ignore]`-by-default integration test that runs the real fetch and
asserts the fetched seed byte-equals the stored seed and the derived
public-key fingerprint matches:

```bash
KERNEL_KEY_GCP_PROJECT=my-project \
KERNEL_KEY_GCP_SECRET=safety-kernel-signing-key \
GCP_KEY_TEST_EXPECT_SEED_B64="$(gcloud secrets versions access latest --secret=safety-kernel-signing-key)" \
  cargo test -p qorch-safety-kernel --test gcp_key_backend_live -- --ignored
```

## Threat model (key at rest)

| Concern | `env` (dev) | `gcp` (prod) |
|---------|-------------|--------------|
| Seed in process environment | yes — readable from `/proc/<pid>/environ`, leaks into core dumps & crash reports | **no** — never placed in the environment |
| Seed at rest | wherever the operator put the env var (shell history, compose file, systemd `Environment=`) | GCP Secret Manager, encrypted at rest, access-logged |
| Runtime identity rights | n/a | read-only (`secretAccessor`) on one secret |
| Rotation | manual env edit + restart | `gcloud secrets versions add` + restart (pin a version to stage rollout) |

**Out of scope for this slice (tracked separately):** dual-sign overlap
windows for zero-downtime rotation and the `KERNEL_KEY_REVOKED` flag
(ARY-1886 Step-14R rotation/revocation); the HSM/TPM signer abstraction
where the private key never leaves the device (ARY-2520 / ARY-2521); and
non-GCE GCP ADC via service-account JSON keys.
